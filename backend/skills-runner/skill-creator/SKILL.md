# 🧬 Skill Creator — 技能自动创建

## 概述

每次完成复杂任务后，系统性地分析工作过程，提取可复用的模式、流程和决策逻辑，将其封装为标准化的 OpenClaw Skill（SKILL.md），实现知识的沉淀与复用。

## 触发时机

| 触发方式 | 说明 |
|---------|------|
| **任务完成后** | 当一个多步骤任务或涉及多个子任务的工作被 CEO 验收通过后 |
| **手动触发** | CEO 明确要求"把这个过程做成技能" |
| **批量扫描** | 每周一次定期审查已完成任务，看是否有技能提炼价值 |

## 工作流程

### Step 1: 回顾任务上下文

回顾刚完成的任务，理解：
- 任务目标是什么
- 涉及哪些子任务和员工
- 每个员工做了什么
- 遇到了什么问题和解决方案
- 最终产出和决策记录

### Step 2: 识别可复用的模式

从任务中提取可技能化的要素：

| 要素 | 问题 | 示例 |
|------|------|------|
| **工作流** | 有没有一个固定的流程可以标准化？ | "安全审计流程 = 风险评估 → 代码审计 → 基线检查 → 加固方案" |
| **检查清单** | 有没有反复出现的检查项？ | "部署检查: config验证→备份→灰度→监控" |
| **决策树** | 有没有判断逻辑可以固化？ | "API Key泄露风险等级: 明文>环境变量>加密存储>密钥管理服务" |
| **模板** | 有没有产出模板可以复用？ | "安全审计报告模板、验收报告模板" |
| **最佳实践** | 有没有总结出做事原则？ | "路径遍历防护三原则: resolved > normalize > startsWith" |

### Step 3: 生成 Skill 骨架

```markdown
---
name: <skill-name>
description: <简短描述>
---

# <标题> — 技能说明

## 概述

## 适用场景

## 核心工作流

### Step 1: 
### Step 2: 
### Step 3: 

## 检查清单

- [ ] 
- [ ] 
- [ ] 

## 参考

- 来源任务: <task-id>
- 创建时间: <date>
```

### Step 4: 验证 Skill

在保存前确认：
- [ ] 名称是否直观（能一眼看出用途）
- [ ] 描述是否准确（能在技能列表中快速定位）
- [ ] 步骤是否完整（不需要额外信息就能执行）
- [ ] 检查清单是否可操作（每条都能打勾/叉）
- [ ] 示例是否已脱敏（不包含真实 API Key、密码等）

### Step 5: 保存并启用

```bash
安装到工作空间技能目录:
~/.openclaw/workspace/skills/<skill-name>/SKILL.md

启用:
openclaw config set skills.entries.<skill-name>.enabled true
```

## 输出示例

```markdown
---
name: security-hardening-checklist
description: 安全加固标准流程 — 代码审计、凭证管理、基线加固一体化
---

# 🔒 安全加固标准流程

## 适用场景
完成代码安全审计后需要输出系统级加固方案时。

## 核心工作流
### Step 1: 凭证安全审查
### Step 2: Web安全加固
### Step 3: 代码安全修复
### Step 4: 基础设施加固
### Step 5: 流程安全建设

## 检查清单
- [ ] API Key 已从明文改为环境变量
- [ ] CORS 已从通配符改为白名单
- [ ] 前端已加认证
- [ ] 路径遍历已防护
- [ ] 日志已脱敏
```

## 注意事项

- 不要创建重复技能（创建前检查是否已存在类似技能）
- 技能内容要通用，不要包含特定项目的硬编码路径
- 技能粒度适中：**太粗**（"软件开发技能"）没有价值，**太细**（"修复xxx文件的第3行"）也没有复用性
