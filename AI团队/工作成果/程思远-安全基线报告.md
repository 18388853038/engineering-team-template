## 2️⃣ 配置基线检查

### 🔴 [CRITICAL] AI Provider API Key 明文存储

- **位置:** `backend/ai-provider.json`
- **问题:** API Key 以明文 JSON 文件存储
- **内容:** `"apiKey": "sk-dab9e24692214b858e129023dcde36c1"`
- **影响:** 任何能读取该文件的人（运维开发、恶意软件、仓库泄露）均可直接使用该 Key，导致 AI 服务被滥用并产生费用
- **建议:**
  1. 立即撤销该 Key 并重新生成
  2. 改用环境变量 `DEEPSEEK_API_KEY`，仅保留 key 名在配置文件中
  3. 将 `ai-provider.json` 加入 `.gitignore`（当前没有 `.gitignore` 文件）

### 🔴 [CRITICAL] 前端 localStorage 存储 API Key

- **位置:** `frontend/index.html` 第 1658 行
- **问题:** 前端代码将 API Key 明文存入浏览器 `localStorage`，Key 名为 `ecs_backup_key`
- **代码:** `localStorage.setItem('ecs_backup_key', d.apiKey);`
- **影响:** XSS 攻击或共用设备可导致 Key 泄露；浏览器 localStorage 无加密保护
- **建议:**
  1. 移除 localStorage 备份逻辑
  2. 前端仅保留「是否已配置 Key」的状态指示，不存储 Key 原文

### 🟠 [HIGH] CORS 配置严重过松

- **位置:**
  - `backend/server-modern.js:1082` — `Access-Control-Allow-Origin: '*'`
  - `backend/server.js:1687` — `Access-Control-Allow-Origin: '*'`
  - 所有 API 响应均允许任意来源
- **问题:** 通配符 `*` 允许任何域名的网页发送跨域请求
- **影响:** 恶意网站可制作 XHR 请求从用户本地访问这些 API，获取或操纵数据
- **建议:**
  1. 禁止使用 `*`，改为白名单特定域名
  2. 若不需跨域，移除 CORS 头或设置为 `Access-Control-Allow-Origin: http://localhost:8003`
  3. 对包含敏感操作的端点（如 `/api/provider/config`）禁用 CORS 或要求带凭证

### 🟠 [HIGH] 缺少 .gitignore / 版本控制风险

- **位置:** 项目根目录及所有子目录
- **问题:** 未发现任何 `.gitignore` 文件
- **影响:** 若启用 Git 版本控制，`ai-provider.json`、`license.json`、日志文件、`.wx-bot-state.json`、`SQLite` 数据库等所有敏感文件都会被直接跟踪并提交
- **建议:** 立即创建 `.gitignore`，至少排除:
  ```
  node_modules/
  backend/ai-provider.json
  backend/license.json
  backend/*.log
  backend/logs/
  backend/*.db
  backend/*.db-shm
  backend/*.db-wal
  .env
  .wx-bot-state.json
  backend/memory-ai_ceo.json
  ```

### 🟠 [HIGH] 多渠道推送凭证未加密存储

- **位置:** `backend/modules/channels.js`
- **问题:** 微信、飞书、钉钉、企微、QQ 机器人的 API 凭证通过环境变量或前端表单传递，但：
  - 无凭证加密存储机制
  - 前端表单输入的 AppSecret/BotToken 以明文通过 HTTP 传输（`http://127.0.0.1:8003`）
  - 凭证可能被记录在日志或 JSONL 文件中
- **影响:** 通信中间人攻击可窃取多渠道推送凭证
- **建议:**
  1. 所有渠道凭证统一通过环境变量注入，不在前端存储
  2. HTTPS 替代 HTTP
  3. 考虑使用 HashiCorp Vault 或类似密钥管理服务

### 🟡 [MEDIUM] ceo_debug.log 泄露会话内容

- **位置:** `backend/ceo_debug.log`
- **问题:** 日志记录了 CEO Agent 与用户的完整对话内容，包括:
  - 用户发送的消息全文
  - API 请求失败时的完整错误信息和 API 响应体
  - 内部 API Key 验证失败的详细信息
- **示例泄露:** `"fallback not OK: 400 {"error":{"message":"The supported API model names are deepseek-v4-pro or deepseek-v4-flash...`
- **影响:** 该日志若被外部访问，将暴露用户隐私对话和系统内部详细信息
- **建议:** 生产环境关闭详细调试日志，或对日志做脱敏处理

### 🟡 [MEDIUM] logs/app.log 记录完整请求路径

- **位置:** `backend/logs/app.log`
- **问题:** 记录了所有 HTTP 请求的来源 IP 和路径，包括疑似泄露的随机文件名（如 `4a53fd4a205a0f4097c2fae42cf0262f.png`）、`.well-known` 探测等
- **建议:** 日志中过滤或脱敏敏感路径，限制日志保留周期

### 🟡 [MEDIUM] stderr.log 泄露系统内部信息

- **位置:** `backend/stderr.log`
- **问题:** 记录了完整的 Node.js 未捕获异常堆栈，包含:
  - 服务器完整路径: `F:\eCompany-OpenClaw\eCompany-OpenClaw\backend\server-modern.js`
  - 端口冲突: `127.0.0.1:8003`
  - 操作系统和 Node 版本信息
- **建议:** 生产环境应捕获异常并记录到受控日志系统，避免堆栈信息暴露

### 🟡 [MEDIUM] 项目包含大量备份文件和旧代码

- **位置:** 项目各目录
- **问题:** 存在大量 `.bak`、`.backup` 文件和遗留脚本（如 `p0.js` ~ `p5.js`、`rebuild_*.js` 等），总计数十个
- **影响:** 旧版本可能包含已废弃的凭证、硬编码凭据或安全漏洞，增加攻击面
- **建议:** 清理无用备份文件，保留必要备份并确保不含敏感信息

### 🟢 [LOW] license.json 激活信息硬编码

- **位置:** `backend/license.json`
- **问题:** 测试激活密钥 `EC-PER-TEST` 和激活时间戳硬编码
- **影响:** 测试密钥泄露影响小，但不应出现在代码库中
