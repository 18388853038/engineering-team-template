## 漏洞1：路径遍历攻击（P0）

**受影响文件：** `server-modern.js`  
**影响范围：** CEO 的 `read_file` / `write_file` 工具函数

### 问题描述
`execCEOTool` 函数中，`read_file` 和 `write_file` 使用 `path.resolve(BASE, args.filepath)` 拼接路径。当攻击者传入以 `/` 开头的绝对路径（如 `/etc/passwd`）或包含 `..` 的相对路径时，`path.resolve` 会解析到 BASE 目录之外，导致任意文件读写。

### 修复方案
在 `read_file` 和 `write_file` 的 `try` 块中，先解析路径，然后检查解析后的路径是否以 `BASE` 开头：

```javascript
var resolvedPath = path.resolve(BASE, args.filepath);
// 路径安全检查：防止路径遍历攻击
if (!resolvedPath.startsWith(BASE)) {
  result = { success: false, message: '不允许访问非项目目录的文件' };
  break;
}
```

### 风险等级
**P0 - Critical** — 可导致服务器任意文件读取（如 `/etc/passwd`、`/proc/self/environ` 等）和任意文件写入（覆盖配置文件、植入恶意代码）。

## 漏洞2：沙箱泄露环境变量（P0）

**受影响文件：** `modules/sandbox.js`  
**影响范围：** `ProcessSandbox._spawnWithTimeout` 方法

### 问题描述
`_spawnWithTimeout` 中，子进程的 `env` 参数为 `{ ...process.env, ...options.env }`，将宿主进程的**所有环境变量**原封不动传入子进程。这意味着 API Keys（如 `DEEPSEEK_API_KEY`、`OPENAI_API_KEY`）、数据库密码、Secret Token 等敏感信息全部暴露给沙箱中执行的任意代码（shell、python、js）。

### 修复方案
新增 `_getSafeEnv(extraEnv)` 方法，创建一个安全的环境变量副本：
- 定义白名单 `SAFE_VARS`：只保留 `PATH`、`HOME`、`TEMP`、`TMP`、`TMPDIR`、`USER`、`USERNAME`、`LANG`、`LC_ALL`、`SHELL`、`TERM` 等无害变量
- 合并调用方显式传入的额外环境变量（`options.env`）
- 在 `_spawnWithTimeout` 中替换为 `this._getSafeEnv(options.env)`

### 风险等级
**P0 - Critical** — 沙箱中的任意代码（包括用户提交的 shell 脚本）可以直接读取所有环境变量，窃取 API Keys 等敏感凭证。
