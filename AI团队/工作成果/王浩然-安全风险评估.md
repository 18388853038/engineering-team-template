**评估人：** 王浩然（CISO）  
**评估日期：** 2026-05-07  
**项目版本：** v2.0.0 (modern)  
**项目根目录：** F:\v3.0_backup_2026-05-05

## 总体评估结论

**安全评级：高危（Critical）** — 存在多处 P0 级严重风险，系统几乎无任何安全防护，生产环境运行存在极大被攻破风险。建议在投入正式使用前，优先修复全部 P0/P1 风险。

## 风险清单

### 🔴 [P0/严重] API Key 明文存储在 `ai-provider.json`

- **位置：** `backend/ai-provider.json`  
- **详情：** DeepSeek API Key `sk-dab9e24692214b858e129023dcde36c1` 以明文JSON格式存储在文件中。任何人只要能读取该文件即可直接使用该API Key，产生费用消耗。
- **影响：** API Key泄露可导致：① 第三方非法调用DeepSeek API产生大量费用 ② 窃取AI对话中的数据 ③ 以系统身份发送恶意请求
- **修复建议：**
  1. 立即吊销该Key并在DeepSeek后台生成新Key
  2. 改用环境变量 `DEEPSEEK_API_KEY` 注入，例如在 `process.env` 或 `.env` 文件中
  3. 移除 `ai-provider.json` 中的 apiKey 字段
  4. 加 `.gitignore` 保护

### 🔴 [P0/严重] 微信 ClawBot Token 明文存储在 `.wx-bot-state.json`

- **位置：** `backend/.wx-bot-state.json`  
- **详情：** 微信ClawBot的完整连接凭据明文存储，包含：
  ```json
  {
    "token": "e0b6c4cfdfb7@im.bot:060000c116287ae8f0e23aa81930a8e2b81b4d",
    "botId": "e0b6c4cfdfb7@im.bot",
    "baseUrl": "https://ilinkai.weixin.qq.com",
    "userId": "o9cq805fMbgiTcfDjgRZraJmIcRM@im.wechat"
  }
  ```
- **影响：** 攻击者可利用此Token以机器人身份发送微信消息、窃取聊天记录、冒充系统与用户交互
- **修复建议：**
  1. 立即吊销该微信Bot的Token
  2. 改用环境变量注入Token（如 `ECOMPANY_WX_ACCESS_TOKEN`）
  3. 添加文件访问权限控制，仅允许Server进程读取
  4. 生产中禁止将此文件提交到版本控制

### 🔴 [P0/严重] 完全无用户认证/鉴权机制

- **位置：** `backend/server-modern.js`（全部API路由）和 `frontend/`  
- **详情：** 系统所有API端点（包括 `/api/chat`, `/api/provider/config`, `/api/sandbox/execute`, `/api/dispatch`, `/api/tasks`, `/api/files` 等）均无任何鉴权——没有JWT Token、没有Session、没有API Key验证、没有任何形式的身份验证。
- **影响：** 任何能够访问该服务器（`127.0.0.1:8003` → 若部署到公网则为所有人）的人均可：
  - 调用所有AI对话接口（消耗API费用）
  - 读写任意文件（文件浏览器路由）
  - 执行任意代码（沙箱接口）
  - 修改AI Provider配置
  - 查看/修改所有任务和团队数据
- **修复建议（分层方案）：**
  1. **短期：** 在HTTP Server层添加简单API Token验证（`Authorization: Bearer <token>`）
  2. **中期：** 实现基于JWT的用户登录注册系统
  3. **长期：** 引入RBAC（基于角色的访问控制），区分管理员/普通用户/只读用户权限
  4. 所有敏感操作（sandbox、文件读写、AI配置修改）增加二次鉴权

---

### 🔴 [P0/严重] 沙箱存在严重安全漏洞，可执行任意代码且无防护

- **位置：** `backend/modules/sandbox.js`（ProcessSandbox）和 `/api/sandbox/execute` 路由  
- **详情：** 沙箱接口存在两个严重问题：
  1. **子进程继承 `process.env`（第111行）：** `env: { ...process.env, ...options.env }` — 将父进程的所有环境变量（包括 API Keys）传入沙箱子进程。编写的恶意脚本可直接通过 `process.env.DEEPSEEK_API_KEY` 读取API Key。
  2. **支持 shell/bash 语言执行（第67行）：** 可执行任意Shell命令，无任何白名单限制
  3. **无鉴权的代码执行接口：** POST `/api/sandbox/execute` 允许任何人提交任意代码执行
- **影响：** 攻击者可获取服务器上的全部敏感数据（API Keys、数据库、文件），在服务器上执行任意命令
- **修复建议：**
  1. **不能将 `process.env` 传递给沙箱子进程**，应仅传递最小必要环境变量
  2. 限制shell执行能力（移除bash/shell执行模式，或添加严格命令白名单）
  3. 生产环境禁用沙箱接口，或仅允许管理员使用
  4. 添加Docker隔离执行（DockerSandbox现有接口但未实现）
  5. 添加CPU/内存使用限制

### 🟠 [P1/高] CORS 配置为完全开放（`Access-Control-Allow-Origin: *`）

- **位置：** `backend/server-modern.js` 第402行 `'Access-Control-Allow-Origin': '*'`，以及第1082-1085行 `res.setHeader('Access-Control-Allow-Origin', '*')`  
- **详情：** 服务端同时存在两处CORS配置，均设置为 `*`（允许任何来源的跨域请求）。任意第三方网站发起的请求均可访问该服务器API。
- **影响：** 结合无认证机制，只要用户浏览器访问过恶意网站，该网站即可通过用户的同网络环境向后端发起任意请求（CSRF攻击）。
- **修复建议：**
  1. 将 `Access-Control-Allow-Origin` 改为明确的域名白名单（生产环境URLs）
  2. 如果不是必须的，移除重复的CORS配置（第402行和1082-1085行重复设置）
  3. 添加CSRF Token防护

---

### 🟠 [P1/高] API Key 在前端 localStorage 明文持久化

- **位置：** `frontend/index.html`  
  - 第1658行：`localStorage.setItem('ecs_backup_key', d.apiKey)` 备份API Key到localStorage
  - 第1661行：`localStorage.setItem('ecs_backup_key', v.trim())` 再次备份
  - 第1654行：`localStorage.getItem('ecs')` — 整个provider配置存储在localStorage
  - 第1676行：`localStorage.setItem('ecompany-config', ...)` — 配置持久化
  - 第1940行：`localStorage.setItem('ec_channel_configs', ...)` — 渠道凭据（含AppSecret、BotToken等）
- **详情：** 前端将用户输入的API Key和渠道凭据（飞书AppSecret、Telegram Bot Token、钉钉ClientSecret、企微AgentSecret等）全部明文存储在浏览器localStorage中。
- **影响：** 任何浏览器扩展、同源恶意脚本、或者共享同一浏览器的其他用户均可读取这些凭据
- **修复建议：**
  1. 禁止前端持久化API Key到localStorage，仅保存在内存中
  2. API Key的管理应仅由后端负责（后端从环境变量读取）
  3. 渠道凭据同样应仅在后端管理，前端只做临时展示
  4. 考虑使用HttpOnly Cookie替代localStorage存储敏感数据

### 🟠 [P1/高] HTTP 明文传输 API Key

- **位置：** `frontend/index.html` 第1659行 `syncProviderToBackend()` 和 `backend/server-modern.js` 第742行 `/api/provider/config`  
- **详情：** 前端通过 HTTP （`http://127.0.0.1:8003`）明文传输 API Key 到后端。如果部署到非本地环境，或本地网络存在中间人攻击，API Key在传输过程中可被抓包截获。
- **影响：** 明文传输敏感凭据，违反安全最佳实践
- **修复建议：**
  1. 所有API通信必须使用HTTPS
  2. 配置TLS证书（Let's Encrypt 免费证书）
  3. 部署到生产环境时配置Nginx反向代理做TLS终止

### 🟡 [P2/中] CEO Tool 中的 read_file/write_file 存在路径遍历风险

- **位置：** `backend/server-modern.js` 第231-243行 `case 'read_file'` 和 `case 'write_file'`  
- **详情：** CEO工具的 `read_file` 和 `write_file` 使用 `path.resolve(BASE, args.filepath)` 解析文件路径。虽然BASE是backend目录，但未充分校验遍历行为（例如传入 `../../../../etc/passwd` 可能绕过限制，取决于BASE路径深度）。
- **影响：** AI CEO可通过工具读取/写入不受允许的文件
- **修复建议：**
  1. 添加 `startsWith` 校验：`if (!resolved.startsWith(BASE)) throw new Error('路径越界')`
  2. 限制读写范围为特定子目录（如 `workspace/`, `data/`）
  3. 记录所有文件读写操作的审计日志

---

### 🟡 [P2/中] 文件浏览器路由存在路径遍历风险

- **位置：** `backend/server-modern.js` 第837-870行 (`/api/files`, `/api/files/read`, `/api/files/write`)  
- **详情：** 虽然代码中有一行 `if (!targetDir.startsWith(BASE))` 检查，但 `path.resolve` 和 `path.join` 在Windows上可能因为路径大小写、符号链接（junction points）、或 `\\?\` 前缀等方式绕过。
- **影响：** 攻击者可能读取/写入服务器上任意文件
- **修复建议：**
  1. 使用 `path.resolve` + `fs.realpathSync` 规范化路径后再做startsWith校验
  2. 添加白名单目录限制
  3. 生产环境禁用文件读写API的写入功能

---

### 🟡 [P2/中] 渠道凭据明文存储在 localStorage

- **位置：** `frontend/index.html` 第1925-1940行（`ec_channel_configs`）和第2080行  
- **详情：** 渠道配置持久化到localStorage，包含飞书AppSecret、钉钉ClientSecret、微信Token、Telegram Bot Token等敏感凭据。这些凭据明文存储，浏览器扩展和恶意脚本可读取。
- **影响：** 聊天渠道的完整访问权限泄露
- **修复建议：**
  1. 渠道凭据应仅在后端存储，前端只展示脱敏版本
  2. 后端应使用加密存储渠道凭据
  3. 移除前端localStorage中的渠道凭据持久化

---

### 🟡 [P2/中] locks.json 锁机制缺乏超时和死锁恢复

- **位置：** `backend/locks.json`  
- **详情：** 锁记录仅有 `active: true/false` 和 `lockedAt` 字段，没有TTL/超时机制。如果获取锁的进程崩溃或未释放，该锁将永久占用，其他进程永远无法获得资源。
- **示例发现：** 存在一条 `"resourceId": "r1"` 的锁已经 `active: false` 但记录仍然保留未被清理。同时存在其他 `active: true` 的锁无到期时间。
- **影响：** 资源死锁导致任务卡死，系统需要手动重启清理
- **修复建议：**
  1. 添加 `ttl`（存活时间）字段，自动过期
  2. 后端定时检查锁过期并自动释放
  3. 添加强制释放锁的API（需管理员权限）
  4. 每次锁申请时清理过期锁

### 🟡 [P2/中] 日志文件可能泄露敏感信息

- **位置：** `backend/ceo_debug.log`、`backend/ceo_error.log`、`backend/server_err.log`、`backend/server_out.log`  
- **详情：** 日志文件包含系统调试信息，如 `ceo_debug.log` 中记录了CEO API调用情况。`server-modern.js` 中多处将错误信息直接返回客户端（如第450-455行和第474行），且API调用的Authorization头可能被日志记录。
- **影响：** 日志文件积累可能暴露API Key、系统内部结构、用户对话内容
- **修复建议：**
  1. 禁止在日志中记录敏感数据（拦截 `Authorization` 头、API Keys）
  2. 添加日志轮转（rotation）和自动清理
  3. 错误信息不要直接返回客户端，使用通用错误消息

### 🟢 [P3/低] 大量旧备份文件残留在生产目录

- **位置：** `backend/server.js`, `backend/server.js.bak`, `backend/server.js.bak.pre-wx-integration`, `backend/server-modern.js.bak`, `backend/server-modern.js.backup.catch-fix`  
- **详情：** 存在多个旧版本备份文件（`.bak`, `.backup`, `bak.pre-wx-integration`等），增加了攻击面和维护混淆的几率。共保留6个版本的服务器文件。
- **影响：** 低风险。但增加了信息泄露面，且容易误修改到旧文件
- **修复建议：**
  1. 清理旧备份文件，只保留当前活跃的 `server-modern.js`
  2. 使用Git进行版本控制替代本地备份文件

---

### 🟢 [P3/低] 无速率限制（Rate Limiting）/请求频率控制

- **位置：** 全部API端点  
- **详情：** 所有API端点均未实现请求频率限制。对话接口（`/api/chat`）可直接被大量调用消耗API费用。
- **影响：** 低风险。但可能导致API Key消耗过速、服务器负载过高
- **修复建议：**
  1. 添加速率限制中间件（如 `express-rate-limit` 或自定义实现）
  2. 对 `/api/chat` 和 `/api/sandbox/execute` 设置严格的调用频率限制（如 10次/分钟）
  3. 添加API调用配额管理

---

### 🟢 [P3/低] 静态文件服务范围过大

- **位置：** `backend/server-modern.js` 第1098-1108行  
- **详情：** 静态文件服务遍历 `FRONTEND` 和 `DIST` 两个目录，如果包含敏感文件（如备份文件、配置文件、.env文件）可能被直接访问。
- **影响：** 低风险。但需确认frontend目录中不包含敏感文件
- **修复建议：**
  1. 严格限制静态文件目录为 `dist/` 一个目录
  2. 添加静态文件访问黑名单（排除 `.json`, `.bak`, `.*` 等文件）

---

### 🟢 [P3/低] 缺少请求体大小限制

- **位置：** `backend/server-modern.js` 第452行 `parseBody()` 函数  
- **详情：** `parseBody` 没有限制请求体大小，攻击者可发送超大请求体导致服务器内存耗尽（DoS攻击）。
- **影响：** 低风险。可能导致服务不可用
- **修复建议：**
  1. 在 `parseBody()` 中添加 `chunk` 累计大小检测，超过限制直接拒绝
  2. 设置合理的大小上限（如 1MB 或 5MB）
