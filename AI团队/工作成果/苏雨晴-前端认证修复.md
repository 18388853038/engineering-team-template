## 漏洞1：前端无任何认证

### 问题描述
打开前端页面就能直接操作所有功能，完全无登录验证，任何人只要有网络访问权限即可使用系统全部功能。

### 修复方案
在 `F:\v3.0_backup_2026-05-05\frontend\activation.js` 中增加了完整的 Token 认证系统：

#### 新增功能
1. **Token 登录界面**（`showTokenLogin` 函数）
   - 美观的暗色主题登录页
   - 密码输入框（隐藏输入）
   - 支持 Enter 键提交
   - 友好的错误提示和状态反馈

2. **Token 验证**（`verifyToken` 函数）
   - 向后端 `POST /api/auth/verify` 发送 Token
   - 验证通过后保存到 `sessionStorage`
   - 验证失败时显示错误信息
   - 防止暴力尝试（提交后禁用按钮）

3. **令牌管理**
   - `getToken()` - 从 sessionStorage 读取 Token
   - `setToken(token)` - 保存 Token 到 sessionStorage
   - `clearToken()` - 清除 Token（登出）
   - `isAuthenticated()` - 检查是否已认证

4. **API 请求自动携带 Token**
   - `addAuthHeader(xhr)` - 为 XHR 请求添加 `Authorization: Bearer <token>` 头
   - 在 `doActivate()` 的激活请求中已集成

5. **401 自动跳回登录**
   - `handle401(xhr)` - 检查响应状态码为 401 时自动清除 Token 并跳转登录页
   - 在 `doActivate()` 和 `verifyToken()` 中均有集成

6. **认证守卫**
   - `show()` 函数入口自动检查认证状态，未认证时显示登录页
   - `window.showActivation` 接口也进行认证检查
   - nav-footer "激活" 链接点击时先检查认证

### 修改文件
- **`F:\v3.0_backup_2026-05-05\frontend\activation.js`** — v1.0 → v1.1

## 漏洞2：修复后端 CORS

### 问题描述
后端 CORS 设置为 `'Access-Control-Allow-Origin': '*'`，允许任意来源访问，存在严重安全风险。

### 修复方案
修改 `F:\v3.0_backup_2026-05-05\backend\server-modern.js`：

#### 具体改动
1. **在 HTTP 请求处理入口**
   - 原代码：`res.setHeader('Access-Control-Allow-Origin', '*')`
   - 修改为：根据请求 `Origin` 头动态判断，仅允许 `http://127.0.0.1:8003` 和 `http://localhost:8003`
   - 验证逻辑：`['http://127.0.0.1:8003', 'http://localhost:8003']`
   - 若 Origin 不在白名单中，默认返回 `http://127.0.0.1:8003`

2. **在 `json()` 辅助函数中**
   - 原代码：`res.writeHead(status, { 'Content-Type': 'application/json; charset=utf-8', 'Access-Control-Allow-Origin': '*' })`
   - 修改为：移除 CORS 头部，因为主请求处理器已设置，避免重复设置导致冲突

3. **新增 Token 认证 API 端点**
   - 新增 `POST /api/auth/verify` 路由
   - 预置有效 Token：`ecompany-token-2026`, `admin-token-2026`
   - 支持从 `backend/tokens.json` 文件读取自定义 Token（可选）
   - 返回 `{ ok: true/false, message: '...' }`
